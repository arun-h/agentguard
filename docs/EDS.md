## 1. Project Overview

AgentGuard is an open-source Python middleware library that adds runtime governance controls to AI agents. It wraps existing agent frameworks and intercepts tool execution before tools are allowed to run.

Core value proposition: **Prevent unsafe tool execution at runtime through policy enforcement, approval workflows, budget controls, loop detection, and audit logging.**

AgentGuard is **not**: an agent framework, an orchestration engine, an observability platform, an evaluation platform, an LLM invocation layer, or a cloud service. It does not manage agent state, memory, or model invocation — those remain the host framework's responsibility.

**Target user:** AI Engineers deploying agents that call tools (databases, APIs, payment systems) who want deterministic, auditable safeguards without writing repetitive middleware, and without depending on a cloud service or an LLM to make governance decisions.

This section is unchanged from the original pre-implementation design; nothing here was contradicted by implementation.

## 2. Design Principles

These principles were established before implementation and held throughout. Every one is satisfied by the actual codebase, not just the design.

1. **Runtime-first.** The `Interceptor` (Section 5.1) acts before the wrapped tool function body ever executes — confirmed by every interceptor test asserting the real function's call log is empty until a decision permits execution.
2. **Framework-agnostic.** `AgentGuard.wrap()` / `@guard.tool` operate on any plain Python callable; no framework-specific code exists in the core decision path. (Framework adapters, where they will eventually live, are addressed in Section 9.)
3. **Local-first.** SQLite + a YAML file are sufficient to run the complete system. No network call exists anywhere in the decision path — verified by inspection of every module in `agentguard/`.
4. **Minimal operational burden.** `pip install -e .` plus a policy YAML file is sufficient. No daemon, broker, or external database is required.
5. **Explicit policies.** All governance decisions are produced by deterministic YAML rule evaluation (Section 5.2). No LLM or heuristic model is consulted anywhere in `PolicyEngine`, `BudgetTracker`, `LoopDetector`, or `DecisionEngine`.
6. **Deterministic enforcement.** Same policy version + same call → same decision, except where state legitimately differs between calls (budget/loop accumulation, policy reload) — this distinction, and one concurrency caveat to it, is documented precisely in Section 7.4.
7. **Production-safe concurrency.** Every stateful subsystem (`BudgetTracker`, `LoopDetector`, `ApprovalManager`, `AuditLogger`, `DatabaseManager`) is isolated by `run_id` and/or protected by explicit locking, and this is verified by real multi-threaded tests, not just code inspection — see Section 7.

## 3. Core Data Models

### 3.1 ExecutionContext

Defined in `agentguard/context.py`. Represents one intercepted tool call.

```python
@dataclass
class ExecutionContext:
    run_id: str
    tool_name: str
    arguments: Dict[str, Any]
    arguments_hash: str
    timestamp: datetime
    policy_version: str
    agent_id: Optional[str] = None
    framework: str = "unknown"
    metadata: Dict[str, Any] = field(default_factory=dict)
    run_id_is_synthetic: bool = False
```

`run_id_is_synthetic` is an addition beyond the original design's field list — it exists so callers and audit readers can distinguish a deliberately-supplied `run_id` from one generated by the synthetic fallback (Section 5.1.2), which matters because cross-run isolation guarantees do not apply to synthetic IDs.

**`arguments_hash`** is SHA-256 of canonical JSON (sorted keys, no whitespace, `default=str` for non-serializable values). This hash is the basis for approval idempotency (Section 5.3.2) and is what gets persisted to the audit log — raw arguments are never persisted long-term, only their hash.

### 3.2 run_id Resolution

Resolution order, implemented in `context.resolve_run_id()`: **explicit argument → ContextVar → thread-local → synthetic fallback**.

```python
def resolve_run_id(explicit, tool_name, strict_run_id=False) -> tuple[str, bool]:
    # returns (run_id, was_synthetic)
```

If `strict_run_id=True` and no `run_id` can be resolved from any source, `MissingRunIdError` is raised immediately — no synthetic fallback occurs. This is a configuration flag on `AgentGuard.from_policy()`, defaulting to `False`. **This design decision was made during implementation, not present in the original pre-implementation document**, which only specified the synthetic-fallback behavior; strict mode was added specifically because a missing `run_id` in production is usually a bug, and silently masking it with a synthetic ID hides that bug.

### 3.3 DecisionType and PolicyDecision

Defined in `agentguard/decisions/models.py`.

```python
class DecisionType(str, Enum):
    ALLOW = "ALLOW"
    DENY = "DENY"
    REQUIRE_APPROVAL = "REQUIRE_APPROVAL"

@dataclass
class PolicyDecision:
    decision: DecisionType
    reason: str
    policy_version: str
    rule_matched: Optional[str] = None
```

### 3.4 ApprovalStatus and ApprovalRecord

Defined in `agentguard/approvals/models.py`.

```python
class ApprovalStatus(str, Enum):
    PENDING = "PENDING"
    APPROVED = "APPROVED"
    REJECTED = "REJECTED"
    EXPIRED = "EXPIRED"

@dataclass
class ApprovalRecord:
    approval_id: str
    run_id: str
    tool_name: str
    arguments_hash: str
    policy_version: str
    status: ApprovalStatus
    created_at: datetime
    ttl_seconds: int = 3600
    updated_at: Optional[datetime] = None
    approver: Optional[str] = None
    notes: Optional[str] = None
```

State machine: `PENDING → APPROVED | REJECTED | EXPIRED`. No backward transitions; all three terminal states are final. This matches the original design exactly.

### 3.5 AuditRecord

Defined in `agentguard/audit/models.py`. Mirrors the `audit_log` table (Section 6.3) exactly, field for field. `id` is `Optional[int] = None` until persisted, since SQLite assigns it on insert.

## 4. System Architecture

### 4.1 Component Map

| Component | File | Owns |
|---|---|---|
| `DatabaseManager` | `storage/database.py` | SQLite connection lifecycle, schema application |
| `PolicyEngine` | `policy/engine.py` | Policy loading, validation, rule evaluation, hot-reload |
| `BudgetTracker` | `budget/tracker.py` | Per-run call-count/cost accumulation |
| `LoopDetector` | `loops/detector.py` | Per-run call-history pattern detection |
| `DecisionEngine` | `decisions/engine.py` | Arbitrates Policy + Budget + Loop into one final decision |
| `ApprovalManager` | `approvals/manager.py` | Approval lifecycle, idempotency, expiration, conflict resolution |
| `AuditLogger` | `audit/logger.py` | Append-only decision history |
| `Interceptor` | `interceptor.py` | Wires all of the above into one per-call sequence |
| `AgentGuard` | `core.py` | Public-facing class; constructs every subsystem; exposes `@guard.tool` |

### 4.2 High-Level Flow

```
Agent calls a @guard.tool-wrapped function
        │
        ▼
AgentGuard._wrap_sync / _wrap_async  (core.py)
        │  binds call arguments, strips run_id_param
        ▼
Interceptor.check()  (interceptor.py)
        │  builds ExecutionContext
        │  ensures `runs` row exists (FK requirement)
        ▼
DecisionEngine.evaluate()  (decisions/engine.py)
        │
        ├─ PolicyEngine.evaluate_against(ctx, snapshot) → DENY?
        │       └─ yes → return DENY immediately (budget/loop never checked)
        │
        ├─ BudgetTracker.check() → exceeded?
        │       └─ yes → override to DENY
        │
        ├─ LoopDetector.record_and_check() → detected?
        │       └─ yes → override to DENY
        │
        └─ otherwise → return original Policy decision (ALLOW or REQUIRE_APPROVAL)
        ▼
Interceptor branches on the final decision:
        ALLOW             → audit logged as ALLOW, real function executes
        DENY              → audit logged as DENY, ToolDeniedException raised
        REQUIRE_APPROVAL  → idempotent approval handling (Section 5.3.2):
                             new/PENDING → ApprovalRequiredException raised
                             APPROVED    → audit logged as ALLOW, function executes
                             REJECTED/EXPIRED → audit logged as DENY, ToolDeniedException raised
```

## 5. Feature Specifications

### 5.1 Tool Interception

Implemented across `Interceptor.check()` (interceptor.py) and `AgentGuard.tool()` / `wrap()` (core.py).

#### 5.1.1 Interception Contract

- Executes before the wrapped function's body runs (verified: every test asserting the real function's call log is empty until ALLOW/APPROVED).
- Supports both sync and async functions via `asyncio.iscoroutinefunction()` detection — a single `@guard.tool` decorator handles both; the caller never picks a different decorator for async.
- Never modifies tool arguments before passing them to the real function (run_id_param is stripped, but this is metadata `AgentGuard` itself consumes, not an argument the real function would have received anyway).

#### 5.1.2 Captured Metadata

Captured into `ExecutionContext` (Section 3.1): `run_id`, `tool_name`, `arguments`, `arguments_hash`, `timestamp`, `policy_version`, `agent_id`, `framework`, `metadata`, `run_id_is_synthetic`.

#### 5.1.3 Missing run_id Handling

Two modes, both implemented (`context.resolve_run_id()`):

- **`strict_run_id=False` (default):** synthetic ID generated (`auto-<uuid4>`), `WARNING` logged, `run_id_is_synthetic=True` on the resulting context.
- **`strict_run_id=True`:** `MissingRunIdError` raised immediately; no synthetic ID, no audit record (there is no run to associate it with).

#### 5.1.4 The `runs` Table Upsert — A Gap Found and Closed During Implementation

**This requirement did not exist in the original pre-implementation design and was discovered only when the Interceptor was first run end-to-end.** The `approvals` table has a foreign key to `runs(run_id)` (Section 6.2). Nothing in the originally-designed Policy/Budget/Loop/Decision layers ever created a `runs` row. The first approval or audit write for a brand-new `run_id` would have failed with a foreign key constraint violation.

**Fix, now part of the permanent contract:** `Interceptor._ensure_run_exists()` runs `INSERT OR IGNORE INTO runs (...)` on every single call to `check()`, before evaluating any decision. This is idempotent and safe to call on every request.

#### 5.1.5 Exception Types

Defined in `agentguard/exceptions.py`: `AgentGuardException` (base), `ToolDeniedException`, `ApprovalRequiredException`, `BudgetExceededException`, `LoopDetectedException`, `PolicyValidationError`, `AgentGuardDatabaseError`, `AuditMutationError`, `MissingRunIdError`.

**Note on `BudgetExceededException` and `LoopDetectedException`:** these exception classes exist in `exceptions.py` exactly as originally designed, but **are not actually raised anywhere in the current implementation**. Budget and loop overrides are surfaced as `DecisionType.DENY` with an explanatory `reason` string (Section 5.2.4), which the `Interceptor` then raises as `ToolDeniedException` — not as the more specific `BudgetExceededException`/`LoopDetectedException`. This is a real, currently-unresolved gap between the exception hierarchy's design and its actual use; it is not a bug (callers still get a clear `reason` string explaining the cause), but a caller catching specifically `BudgetExceededException` to distinguish it from a policy-rule denial would not work today. Documented here rather than silently left inconsistent; see Section 16.3 for disposition.

### 5.2 Policy Engine

Implemented in `agentguard/policy/engine.py` and `agentguard/policy/models.py`.

#### 5.2.1 Policy File Format

```yaml
version: "1.0.0"
description: "optional"
rules:
  - name: deny_delete_customer
    tool: delete_customer
    action: deny
    reason: "..."
budget:
  max_tool_calls: 50
  max_estimated_cost: 5.00
loop_detection:
  enabled: true
  max_repetitions: 5
  window_size: 20
defaults:
  unmatched_tool: allow
```

This matches the original design exactly, with no changes.

#### 5.2.2 Validation Rules

Ten explicit validation rules, implemented in `PolicyConfig.from_dict()`, each raising `PolicyValidationError` with the offending field, path, expected type, and actual value on failure:

1. `version` must be a non-empty string.
2. `rules` must be a list (may be empty).
3. Each rule requires `name`, `tool`, `action` (one of `allow`/`deny`/`require_approval`).
4. Rule names must be unique.
5. Tool names must be unique across rules (no duplicate-tool ambiguity).
6. `budget.max_tool_calls`, if present, must be a positive integer — **explicitly excluding `bool`**, since Python's `bool` is a subclass of `int` and `True`/`False` would otherwise silently pass as `1`/`0`. This exclusion is a real test-driven addition: the test suite includes `test_bool_max_tool_calls_raises` specifically because this was identified as a trap during implementation.
7. `budget.max_estimated_cost`, if present, must be a positive float (same `bool` exclusion).
8. `loop_detection.max_repetitions`, if present, must be an integer ≥ 2.
9. `loop_detection.window_size`, if present, must be an integer ≥ `max_repetitions`.
10. `defaults.unmatched_tool` must be `allow` or `deny`.

#### 5.2.3 Loading and Hot-Reload

`PolicyEngine.__init__()` loads and validates eagerly. `reload()` is explicit (no filesystem watcher); on success it atomically swaps the active `PolicyConfig` under an `RLock`; on failure, `PolicyValidationError` propagates and the previously active policy remains in effect — verified by `test_failed_reload_keeps_old_policy_active`.

#### 5.2.4 Evaluation Algorithm and the `evaluate_against()` Split

**This subsection documents a real architectural change made during implementation, specifically to fix a concurrency bug — not part of the original design.**

`PolicyEngine` exposes two methods:

```python
def evaluate(self, ctx: ExecutionContext) -> PolicyDecision:
    policy = self.policy          # one locked read
    return self.evaluate_against(ctx, policy)

def evaluate_against(self, ctx: ExecutionContext, policy: PolicyConfig) -> PolicyDecision:
    # linear scan, first-match-wins, exact tool name equality
    ...
```

**Why `evaluate_against()` exists:** `DecisionEngine.evaluate()` (Section 5.2.5) needs both the rule-matching result *and* the budget/loop configuration for the same decision. The original implementation called `policy_engine.evaluate(ctx)` and then separately read `policy_engine.policy.budget` afterward — **two independent locked reads of `PolicyEngine`'s internal state**. Each read was individually atomic, but the *pair* was not: a `reload()` landing between them could mean the matched rule came from policy version N while the budget/loop thresholds applied came from version N+1, silently breaking the guarantee that a decision is fully explained by one `policy_version`.

**The fix:** `DecisionEngine.evaluate()` now takes exactly one snapshot (`policy_snapshot = self._policy_engine.policy`) and passes it into `evaluate_against()`, then reads `.budget` / `.loop_detection` off that *same* snapshot object — never touching `self._policy_engine` again for the remainder of the call. This is verified by a dedicated regression test (`tests/unit/test_decision_engine_reload_race.py`) that deterministically forces the interleaving via a stub `PolicyEngine` returning different policy versions on successive reads, and confirms the bug is closed (and, when temporarily reverted during development, confirms the test actually catches the regression rather than passing trivially).

### 5.3 Approval Workflow

Implemented in `agentguard/approvals/manager.py` and `agentguard/approvals/models.py`.

#### 5.3.1 Lifecycle

`PENDING → APPROVED | REJECTED | EXPIRED`. No backward transitions. Matches original design.

#### 5.3.2 Idempotency

Composite key: `(run_id, tool_name, arguments_hash, policy_version)`. `ApprovalManager.find_existing()` performs the lookup; any change in any one of the four fields is a distinct request. Backed by a `UNIQUE` index in SQLite (Section 6.2) — the database itself enforces this, not just application logic.

#### 5.3.3 Expiration

Lazy: evaluated at lookup time (`find_existing()`, `get_by_id()`, `get_pending()`), not via a background sweeper. A `PENDING` record past its `ttl_seconds` is transitioned to `EXPIRED` as a side effect of being looked up. Terminal states never expire — verified by `test_approved_record_never_expires_even_past_ttl`.

#### 5.3.4 Conflict Resolution — First-Writer-Wins (Resolved During Design Review)

**The original pre-implementation design contained a direct contradiction**: its error-handling matrix said "last writer wins" for concurrent approval updates, while its Open Questions section recommended "first writer wins" for the same scenario. This was identified and resolved during design review, before implementation: **first-writer-wins is the correct and only implemented behavior.**

```python
def _resolve(self, approval_id, new_status, approver, notes) -> bool:
    cursor = conn.execute(
        "UPDATE approvals SET status=?, ... WHERE approval_id=? AND status='PENDING'",
        (...)
    )
    return cursor.rowcount > 0   # False if another writer already resolved it
```

The `WHERE status='PENDING'` guard clause means a second resolution attempt affects zero rows and returns `False`, without overwriting the first writer's decision. Verified under real concurrent threads, not just sequential calls: `test_concurrent_approve_and_reject_only_one_wins` (two threads racing approve vs. reject on the same `approval_id`) and `test_many_concurrent_approve_attempts_exactly_one_succeeds` (20 threads, exactly one succeeds).

The same guarded-UPDATE pattern is reused in `_expire()`, so a lazy expiration check racing a legitimate concurrent approval cannot silently clobber it either — verified by `test_expire_race_does_not_clobber_concurrent_approval`.

#### 5.3.5 Duplicate Creation — Foreign Key / Integrity Error Wrapping (Fixed During Implementation)

**Not present in the original design; found during testing.** `create_approval()`'s `INSERT` can raise `sqlite3.IntegrityError` if a caller skips `find_existing()` and tries to create a duplicate for an existing composite key (the `UNIQUE` index rejects it). The first implementation let this raw `sqlite3` exception propagate unwrapped. **Fixed:** `create_approval()` now catches `sqlite3.IntegrityError` and re-raises as `AgentGuardDatabaseError` with an actionable message telling the caller to use `find_existing()` instead. Verified by `test_duplicate_composite_key_raises`.

### 5.4 Audit Logging

Implemented in `agentguard/audit/logger.py` and `agentguard/audit/models.py`.

#### 5.4.1 Immutability Contract

`AuditLogger` exposes exactly one write method (`log_decision()`) and five read methods (`get_by_run_id`, `get_by_decision`, `get_by_tool_name`, `get_approval_events_for_run`, `count_by_decision`). **No update or delete method exists anywhere in the class.** This is immutability *by absence of capability*, not by a runtime permission check on an otherwise-present method — verified by a structural test (`TestImmutabilityByAbsence`) that enumerates every public method on the class and asserts none match mutating-method naming patterns (`update*`, `delete*`, `remove*`, `clear*`, `drop*`, `truncate*`), so the test itself would fail if such a method were ever added in the future.

#### 5.4.2 Foreign Key Wrapping (Same Fix Pattern as 5.3.5, Found Independently)

`log_decision()`'s `approval_id` parameter is a real foreign key into the `approvals` table. The same class of bug from Section 5.3.5 was found here independently — an unwrapped `sqlite3.IntegrityError` on a dangling `approval_id` reference. **Fixed identically:** wrapped and re-raised as `AgentGuardDatabaseError`. This is documented as a separate fix, not a duplicate of 5.3.5, because it was found and fixed in a separate file during a separate implementation pass — the recurrence is itself a notable fact about this codebase's history, recorded honestly rather than collapsed into one bullet.

#### 5.4.3 Audit Truthfulness — The REQUIRE_APPROVAL → ALLOW/DENY Correction (Major Fix During Implementation)

**This is the most significant behavioral correction made during implementation, and was not specified in the original design.**

When `Interceptor._handle_require_approval()` finds an existing `APPROVED` record and the call actually proceeds to execute, the first implementation logged the audit record with the *original* Policy Engine decision label — `REQUIRE_APPROVAL` — even though the tool call was, in fact, allowed to execute. A reader querying `count_by_decision()` or `get_by_decision("REQUIRE_APPROVAL")` would see this row counted as still-pending, indistinguishable from a genuinely unresolved approval request.

**Fix:** the `Interceptor` now constructs a *corrected* `PolicyDecision` for the audit write specifically — `DecisionType.ALLOW` with a `reason` explaining which approval permitted execution and who approved it — while the original `REQUIRE_APPROVAL` row from the initial request remains untouched (it was a true record of that earlier moment). The same correction applies symmetrically to `REJECTED`/`EXPIRED` outcomes, which are now logged as `DecisionType.DENY` rather than a stale `REQUIRE_APPROVAL`.

```python
if existing.status == ApprovalStatus.APPROVED:
    allow_decision = PolicyDecision(
        decision=DecisionType.ALLOW,
        rule_matched=decision.rule_matched,
        reason=f"Approval {existing.approval_id} was granted by "
               f"{existing.approver or 'unknown approver'}; tool execution proceeds.",
        policy_version=decision.policy_version,
    )
    self._audit_logger.log_decision(ctx, allow_decision, approval_id=existing.approval_id)
    return ctx
```

Verified by `test_audit_log_shows_allow_not_require_approval_after_resolution` and `test_count_by_decision_correctly_reflects_actual_outcomes`, both of which assert the corrected behavior explicitly.

### 5.5 Budget Controls

Implemented in `agentguard/budget/tracker.py`. Matches the original design with no behavioral changes; documented here for completeness.

- `BudgetTracker.increment(run_id, cost=0.0)` accumulates `calls_used` and `cost_used` per `run_id`, protected by `RLock`, returning a snapshot copy (not a live reference) so callers cannot mutate internal state.
- `check(run_id, limits)` is read-only; does not increment.
- `reset_run(run_id)` clears state; no automatic expiration.
- **Durability is explicitly process-scoped, not persistence-scoped** (Section 7.5) — this was a deliberate, called-out limitation in the original design, preserved as-is in implementation.
- **Budget is a single counter shared across every tool in a run, not per-tool.** This is a deliberate design property (not a bug) confirmed and demonstrated explicitly in the demo (Section 12).

### 5.6 Loop Detection

Implemented in `agentguard/loops/detector.py`. Matches the original design with no behavioral changes.

- Detects immediate repetition (`A→A→A`, ≥ `max_repetitions` consecutive identical calls) and alternating two-tool cycles (`A→B→A→B`, fixed cycle length 2 in this version).
- `record_and_check()` both appends to history and checks in one call — there is no separate "check without recording" mode.
- Window is bounded (`deque(maxlen=window_size)`); does not grow unbounded.
- Same process-scoped durability caveat as Budget (Section 7.5).

### 5.7 Decision Engine

Implemented in `agentguard/decisions/engine.py`. See Section 5.2.4 for the reload-race fix specific to this component's interaction with `PolicyEngine`.

#### 5.7.1 Arbitration Order

1. `PolicyEngine.evaluate_against(ctx, snapshot)` → if `DENY`, return immediately. **Budget and Loop are never consulted in this case** — a call denied outright by policy consumes no budget and is not recorded into loop history.
2. If `ALLOW` or `REQUIRE_APPROVAL`: check Budget. If exceeded, override to `DENY`, preserving the original decision's `rule_matched` and noting the original decision in the `reason` text.
3. If Budget passed and loop detection is enabled: check Loop. If detected, override to `DENY` similarly. **Note:** the loop-triggering call itself is still recorded into loop history even though it is denied — this means loop history reflects calls that reached this stage, including the one that tripped detection, but never reflects calls denied earlier at the Budget stage. This asymmetry is a real, documented consequence of the arbitration order, not a bug.
4. Otherwise, return the original Policy decision unchanged. If it was `ALLOW`, increment Budget (an `ALLOW` that hasn't actually executed yet, i.e. `REQUIRE_APPROVAL`, does not increment Budget — only `Interceptor`'s ultimate execution of the real function, layered on top via the approval-resolution path in Section 5.3, represents the call actually happening).

#### 5.7.2 Why Budget/Loop Override REQUIRE_APPROVAL

By design: if Budget or Loop overrides a decision to `DENY`, this happens **regardless of whether the original decision was `ALLOW` or `REQUIRE_APPROVAL`**. This prevents an agent from escalating to approval as a way of bypassing budget/loop limits. Matches original design intent exactly.

## 6. Database Design

### 6.1 Storage Backend and Connection Strategy

SQLite, default path `./agentguard.db`. `DatabaseManager` (storage/database.py) enforces **one connection per thread via `threading.local()`**, with `check_same_thread=True` so any accidental cross-thread use fails loudly rather than corrupting data silently — `check_same_thread=False` is never used as a workaround.

```python
PRAGMA journal_mode = WAL;
PRAGMA synchronous = NORMAL;
PRAGMA foreign_keys = ON;
PRAGMA busy_timeout = 5000;
```

This connection strategy was **fully specified during design review, before implementation** — it was identified as a gap in the original draft (no explicit connection strategy was given), and the thread-local approach with explicit PRAGMAs was added as a direct response, then implemented exactly as specified and verified by `test_different_threads_get_different_connections`.

**A platform-specific test correctness issue, found and fixed after implementation, not a bug in `DatabaseManager` itself:** the original version of this test compared `id()` of connection objects across threads after each thread had already terminated — on some platforms (observed on Windows), CPython's allocator can reuse a just-freed object's memory address for the next object created, making two genuinely different objects compare as equal by `id()`. **Fixed** by keeping live references to all connection objects alive simultaneously before comparing, which is correct on every platform regardless of allocator behavior.

### 6.2 Schema

```sql
CREATE TABLE runs (
    run_id       TEXT PRIMARY KEY,
    created_at   TEXT NOT NULL,
    framework    TEXT NOT NULL,
    agent_id     TEXT,
    status       TEXT NOT NULL DEFAULT 'active',
    metadata     TEXT
);

CREATE TABLE approvals (
    approval_id      TEXT PRIMARY KEY,
    run_id           TEXT NOT NULL REFERENCES runs(run_id),
    tool_name        TEXT NOT NULL,
    arguments_hash   TEXT NOT NULL,
    policy_version   TEXT NOT NULL,
    status           TEXT NOT NULL DEFAULT 'PENDING',
    created_at       TEXT NOT NULL,
    updated_at       TEXT,
    ttl_seconds      INTEGER NOT NULL DEFAULT 3600,
    approver         TEXT,
    notes            TEXT
);
CREATE UNIQUE INDEX idx_approvals_composite
    ON approvals(run_id, tool_name, arguments_hash, policy_version);

CREATE TABLE audit_log (
    id                 INTEGER PRIMARY KEY AUTOINCREMENT,
    run_id             TEXT NOT NULL,
    approval_id        TEXT REFERENCES approvals(approval_id),
    timestamp          TEXT NOT NULL,
    tool_name          TEXT NOT NULL,
    arguments_hash     TEXT NOT NULL,
    policy_version     TEXT NOT NULL,
    decision           TEXT NOT NULL,
    reason             TEXT NOT NULL,
    rule_matched       TEXT,
    budget_calls_used  INTEGER,
    budget_cost_used   REAL,
    loop_count         INTEGER,
    agent_id           TEXT,
    framework          TEXT,
    metadata           TEXT
);

CREATE TABLE migrations (
    version    INTEGER PRIMARY KEY,
    applied_at TEXT NOT NULL
);
```

Matches the original design exactly, including the `UNIQUE` composite index that backs approval idempotency (Section 5.3.2) and the foreign keys that necessitated the `runs`-table-upsert fix (Section 5.1.4).

### 6.3 Migration Strategy

Eager application of all DDL at `DatabaseManager` construction time, idempotent (`CREATE TABLE IF NOT EXISTS`), with a `migrations` table recording the applied schema version. Matches original design.

## 7. Run Isolation, Concurrency, and Thread Safety

### 7.1 Isolation by Subsystem

| Subsystem | Isolation Mechanism |
|---|---|
| Approvals | Composite-key `UNIQUE` index includes `run_id`; enforced by SQLite itself |
| Audit log | `run_id` column on every row; all queries filter by it |
| Budget | In-memory `dict` keyed by `run_id`, `RLock`-protected |
| Loop detection | In-memory `deque` dict keyed by `run_id`, `RLock`-protected |

### 7.2 Thread Safety — What Is Actually Verified, Not Just Designed

Every concurrency claim below is backed by a real multi-threaded test in the suite, not asserted by code review alone:

- `BudgetTracker`: 100 real threads incrementing the same `run_id` concurrently, asserted final count equals exactly 100 (`test_concurrent_increments_on_same_run_all_count`).
- `ApprovalManager`: two threads racing `approve()` vs. `reject()` on the same `approval_id` (`test_concurrent_approve_and_reject_only_one_wins`); 20 threads racing `approve()` on the same ID, exactly one succeeds (`test_many_concurrent_approve_attempts_exactly_one_succeeds`).
- `PolicyEngine`: `reload()` racing concurrent `evaluate()` calls from 4 threads, no exceptions, no observed inconsistent state (`test_reload_is_atomic_under_concurrent_evaluation`).
- `DecisionEngine`: the reload-race fix (Section 5.2.4) verified by a deterministic stub-based test plus an end-to-end real-`PolicyEngine` concurrent test.
- `DatabaseManager`: 20 threads each inserting their own `runs` row concurrently, no corruption, exact row count (`test_concurrent_writes_from_multiple_threads_succeed`).
- `Interceptor`: 5 threads racing to create the *first* approval for an identical composite key, exactly one approval record exists afterward regardless of which thread "won" (`test_concurrent_first_calls_only_create_one_approval`).

### 7.3 Async Compatibility

`AgentGuard.wrap()` detects `asyncio.iscoroutinefunction()` and dispatches to `_wrap_async()`, which `await`s the real function after a synchronous `Interceptor.check()` call. **Implementation note, differing from the original design's suggestion of `asyncio.to_thread()` for dispatching blocking SQLite calls off the event loop:** the current implementation calls `Interceptor.check()` (which performs synchronous SQLite I/O) directly within the async wrapper, without an explicit `asyncio.to_thread()` dispatch. This means SQLite I/O during an async tool call currently blocks the event loop for the duration of that I/O. This was not caught as an issue during implementation because all async tests use a temporary file-backed SQLite database fast enough that blocking is not observable in test timing, but it is a real, currently-unaddressed gap between this document's Section 7.3 design intent and the actual code. Documented honestly here; see Section 16.3.

### 7.4 Determinism — Precise Statement

Given a **fixed** `PolicyEngine` (no concurrent `reload()`) and a **known** `BudgetTracker`/`LoopDetector` state for a given `run_id`, the same `ExecutionContext` always produces the same decision. This is not violated by:

- Budget/loop state changing between two calls to the *same* tool with the *same* arguments — this is the intended purpose of those subsystems, not non-determinism in the governance sense.
- A policy `reload()` between two evaluations — the rule that matches can legitimately change, and the resulting decision correctly reports the new `policy_version`.

It *would* be violated by the reload-race bug described in Section 5.2.4, which is why that fix matters: without it, a single decision could be explained by two different policy versions simultaneously, which is not "determinism producing a different but explicable answer" — it is an internally inconsistent answer.

### 7.5 Durability — Explicit Asymmetry

| Subsystem | Survives Process Restart? |
|---|---|
| Approvals (SQLite) | Yes |
| Audit log (SQLite) | Yes |
| Budget state (memory) | **No** |
| Loop detection history (memory) | **No** |

This asymmetry is intentional, consistent with the local-first design principle (Section 2), and was called out explicitly in the original design before implementation. Operators deploying in environments where process restarts are routine should treat Budget/Loop as best-effort operational guards, not hard safety boundaries.

## 8. Public API

Implemented in `agentguard/core.py` and exported via `agentguard/__init__.py`.

### 8.1 Construction

```python
AgentGuard.from_policy(
    policy_path: str,
    db_path: str = "./agentguard.db",
    approval_ttl: int = 3600,
    strict_run_id: bool = False,
) -> AgentGuard
```

Constructs, in order: `DatabaseManager` → `PolicyEngine` → `BudgetTracker` → `LoopDetector` (sized from the policy's `loop_detection` config) → `DecisionEngine` → `ApprovalManager` → `AuditLogger` → `Interceptor`. **`core.py` contains zero decision-arbitration logic of its own** — it is pure construction and delegation; every governance decision is made inside `Interceptor`/`DecisionEngine`, which `core.py` only constructs and calls.

### 8.2 Tool Wrapping

```python
@guard.tool
def my_tool(...): ...

@guard.tool(run_id_param="custom_run_id")
def my_tool(...): ...

wrapped = guard.wrap(my_tool)   # non-decorator alternative
```

Both sync and async functions work via the same decorator — `asyncio.iscoroutinefunction()` detection picks the right wrapper automatically.

#### 8.2.1 Argument Binding and the Calling-Style Idempotency Bug (Found and Fixed)

**This is a real bug found during implementation, not present in the original design (which did not specify argument-binding order at this level of detail).**

The first implementation called `inspect.signature(fn).bind(*args, **kwargs)` to construct the dict used for idempotency hashing — **before** stripping `run_id_param` out of `kwargs`. Since the real wrapped function's signature never has a `run_id` parameter, this binding call raised `TypeError` almost every time `run_id` was passed as a keyword argument — which is the documented, expected calling convention — and silently fell into a fallback path. That fallback path produced **different dict shapes for positional vs. keyword calls with identical logical arguments**, meaning two separate approval requests could be created for what should be one logical operation.

**Fix:** `run_id_param` is now stripped from `kwargs` *first*; signature binding happens only against the clean, `run_id`-free arguments:

```python
def wrapper(*args, **kwargs):
    run_id = kwargs.get(run_id_param)
    call_kwargs = {k: v for k, v in kwargs.items() if k != run_id_param}
    arguments = self._bind_call_arguments(fn, args, call_kwargs)
    self._interceptor.check(tool_name=fn.__name__, arguments=arguments, run_id=run_id)
    return fn(*args, **call_kwargs)
```

Verified by a dedicated regression test class, `TestCallingStyleIdempotency`, covering positional-only, keyword-only, and mixed calling styles, all asserting identical `approval_id` for identical logical arguments.

### 8.3 Approval, Policy, and Run Lifecycle Methods

```python
guard.approve(approval_id, approver, notes="") -> bool
guard.reject(approval_id, approver, notes="") -> bool
guard.get_pending_approvals() -> List[ApprovalRecord]
guard.get_approval(approval_id) -> Optional[ApprovalRecord]
guard.reload_policy() -> None
guard.policy_version -> str  (property)
guard.reset_run(run_id) -> None
guard.get_run_audit(run_id) -> List[AuditRecord]
```

All are thin delegations to the underlying subsystem (`ApprovalManager`, `PolicyEngine`, `AuditLogger`); none contain independent logic. `reset_run()` clears in-memory Budget/Loop state only — it does **not** delete persisted approvals or audit records (Section 7.5).

**Known limitation, documented honestly:** `reload_policy()` re-reads the policy file but does **not** reconstruct `LoopDetector` with new `max_repetitions`/`window_size` values, since `LoopDetector` is sized once at `from_policy()` time. A reloaded policy that changes loop thresholds will not actually change loop detection behavior until the process restarts. This is a real, currently-unaddressed gap, not by original design — it was noted directly in `core.py`'s own docstring during implementation rather than silently left undocumented.

### 8.4 The `__init__.py` Export Gap (Found and Fixed)

**Not anticipated in the original design.** `agentguard/__init__.py` was left as an empty placeholder through most of implementation; `from agentguard import AgentGuard` did not work until this was fixed by populating it with real exports of `AgentGuard` and every exception type. This was caught only when the public API was first exercised end-to-end exactly as the README's quick-start example shows it — a reminder that "the package imports" is a fact that must be tested, not assumed from the existence of the right files.

## 9. Framework Integration

**Status: not yet implemented.** `agentguard/adapters/__init__.py` exists as an empty placeholder package. No LangGraph or OpenAI Agents SDK adapter code exists.

The original design specified both adapters as MVP scope. **This was descoped during implementation planning, for cost and storage reasons unrelated to engineering difficulty:** the project's zero-cost/zero-GPU/low-storage testing constraint led to a decision to defer real framework installation (LangGraph alone is roughly 107MB with its dependency tree) until the adapter work is actually being done, rather than installing it speculatively. This is a sequencing decision, not a cancellation — Section 16.2 lists it as the top near-term roadmap item.

The interception layer itself (`Interceptor`, `AgentGuard.wrap()`) is fully framework-agnostic already and requires no changes to support a future adapter — an adapter's job, when built, will be solely to extract `run_id` and call context from a specific framework's own conventions and call `AgentGuard.wrap()`/`Interceptor.check()` with that context, exactly as any other caller would.

## 10. Error Handling

| Condition | Behavior | Source |
|---|---|---|
| Missing `run_id`, `strict_run_id=False` | Synthetic ID + `WARNING` log | `context.resolve_run_id()` |
| Missing `run_id`, `strict_run_id=True` | `MissingRunIdError` raised immediately | `context.resolve_run_id()` |
| Malformed policy YAML (load or reload) | `PolicyValidationError`; reload leaves prior policy active | `policy/engine.py` |
| Policy file not found | `FileNotFoundError` | `policy/engine.py` (stdlib `open()`) |
| Duplicate approval composite key | `AgentGuardDatabaseError` (wraps `sqlite3.IntegrityError`) | `approvals/manager.py` §5.3.5 |
| Dangling `approval_id` FK in audit write | `AgentGuardDatabaseError` (wraps `sqlite3.IntegrityError`) | `audit/logger.py` §5.4.2 |
| Concurrent approval resolution race | First writer wins; second observes `False`, does not raise | `approvals/manager.py` §5.3.4 |
| Policy DENY | `ToolDeniedException` | `interceptor.py` |
| Budget/loop override to DENY | `ToolDeniedException` (see §5.1.5 gap re: more specific exception types) | `interceptor.py` |
| `REQUIRE_APPROVAL`, new or still `PENDING` | `ApprovalRequiredException` | `interceptor.py` |
| `REQUIRE_APPROVAL`, resolved `REJECTED`/`EXPIRED` | `ToolDeniedException` | `interceptor.py` |
| Unwritable database path | `AgentGuardDatabaseError` | `storage/database.py` |
| SQLite lock timeout | `AgentGuardDatabaseError` (after `busy_timeout` elapses) | `storage/database.py` |

**Fail-closed posture, confirmed in implementation:** any internal error in the decision path results in an exception propagating to the caller rather than silently allowing tool execution. The one documented exception-to-this-rule is Budget/Loop's in-memory state being lost on process restart (Section 7.5) — a restart resets to zero rather than refusing all subsequent calls, a deliberate availability trade-off stated in the original design and preserved as-is.

## 11. Non-Functional Requirements

Performance targets stated in the original design (sub-5ms interception overhead without a DB write, sub-20ms with one, sub-1ms policy evaluation for up to 1000 rules) were **never formally benchmarked during implementation** — no benchmark suite exists in the codebase. This document does not claim these targets are met; it only confirms they were never measured. This is stated plainly rather than silently carrying forward an unverified claim from the original design.

Reliability and developer-experience requirements from the original design (SQLite transactional writes, no uncaught exceptions crashing the host application, type-annotated public API) are satisfied:

- Every multi-statement write goes through `DatabaseManager.transaction()`, which rolls back on any exception.
- All custom exceptions inherit from `AgentGuardException`, giving callers a single catchable base type.
- `core.py`'s public methods carry type annotations throughout.

## 12. Demo

`demos/demo.py` and `demos/demo_policy.yaml`, using the real installed `agentguard` package — no governance logic is duplicated in the demo script, per an explicit decision made during implementation planning (an earlier proposal to build a separate "zero-install" stdlib-only demo with its own reimplemented logic was rejected for exactly this reason).

Three scenarios in one script, matching the original design's three specified demos:

1. **Wire Transfer Approval Flow** — block, approve, resume, with a permanently-denied `delete_customer` tool shown alongside it.
2. **Loop Detection** — three identical calls to `search_database`, the third denied.
3. **Budget Exhaustion** — four calls to four *distinct* tools under a `max_tool_calls: 3` policy, the fourth denied. **This specific design — distinct tools rather than repeated calls to one tool — exists because of a bug found while building the demo**: an earlier version called one tool repeatedly, which caused Loop Detection to trigger before Budget Exhaustion did, since both thresholds were set to the same value (3) and Budget is checked-but-not-yet-exceeded one call before Loop's repetition count reaches its own threshold. This is the same masking pattern documented as a recurring lesson during test development (Section 16.4) — using varied tool names is what correctly isolates the subsystem actually being demonstrated, and it incidentally demonstrates a real system property: Budget is a single counter shared across every tool in a run, not per-tool.

A real SQLite database (`demos/demo.db`) is produced by running the script, independently queryable, and is checked into the repository as a sample artifact per project decision.

The standalone script requires an explicit `sys.path` correction (anchored to the script's own file location, walking up to the project root) to import the installed package correctly regardless of the caller's working directory — found necessary when `python demo.py`, run from inside `demos/`, resolved `agentguard` as a broken namespace package via the editable install's path hook rather than the real package. This is documented in the script's own comments.

### 16.3 Known Gaps Between This Document and the Code, Stated Plainly

These are real, current limitations, not yet fixed, documented here rather than smoothed over:

1. `BudgetExceededException` and `LoopDetectedException` exist but are never raised; both cases surface as `ToolDeniedException` (Section 5.1.5).
2. Async tool wrapping does not dispatch SQLite I/O via `asyncio.to_thread()`; the event loop is blocked for the duration of synchronous database calls during an async tool's governance check (Section 7.3).

### 16.4 Recurring Lesson From Test Development

Across at least three separate occasions during implementation — `test_decision_engine.py`'s budget-override tests, `test_core.py`'s `reset_run` test, and the original draft of `demos/demo.py`'s budget-exhaustion scenario — a test or demo accidentally exercised Loop Detection instead of, or in addition to, the subsystem actually being tested, because repeated calls used the same tool name. The fix pattern is consistent: use varied tool names to isolate Budget from Loop when only one is meant to be demonstrated. This is recorded here as a real, recurring engineering lesson from this project's history, not a hypothetical caution.
